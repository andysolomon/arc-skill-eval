# Conventional Commits and Semantic Release Plan

## Objective
Set up `arc-skill-eval` to use Conventional Commits and semantic-release, with the release line seeded at `0.1.0` and ready for ongoing automated releases from `main`.

## Current State
- New repo with no existing versioning tooling
- No prior tags or release automation
- Package will be published as an npm package

## Decisions
- Release branch: `main`
- Tag format: `v${version}`
- npm publishing: enabled
- Release notes/changelog: generated by semantic-release
- Initial seed version: `0.1.0`

## Why seed at 0.1.0?
semantic-release normally begins from the latest matching git tag. For a new repo with no release tags, the default first semantic release behavior trends toward `1.0.0`.

To begin the package lifecycle at `0.1.0`, we will:
1. initialize the package at version `0.1.0`
2. install semantic-release and core plugins
3. add release automation on `main`
4. seed the repo with an annotated `v0.1.0` tag once the initial bootstrap commit is ready
5. allow subsequent semantic-release runs to increment from that seed tag

## Phases

### Phase 1 — Repo and package bootstrap
- create `package.json`
- set package name to `arc-skill-eval`
- set package version to `0.1.0`
- add semantic-release script

### Phase 2 — Semantic-release installation and config
- install:
  - `semantic-release`
  - `@semantic-release/commit-analyzer`
  - `@semantic-release/release-notes-generator`
  - `@semantic-release/changelog`
  - `@semantic-release/npm`
  - `@semantic-release/github`
  - `@semantic-release/git`
  - `conventional-changelog-conventionalcommits`
- create `.releaserc.json`
- enable changelog, npm publish, GitHub release publishing, and version file updates

### Phase 3 — GitHub Actions release automation
- add `.github/workflows/release.yml`
- run semantic-release on pushes to `main`
- require `GITHUB_TOKEN`
- require `NPM_TOKEN` for npm publish

### Phase 4 — Seed the release line at `0.1.0`
- after the initial bootstrap commit exists, create annotated tag:

```bash
git tag -a v0.1.0 -m "chore(release): seed 0.1.0"
git push origin v0.1.0
```

This establishes the baseline so future semantic-release runs increment from `0.1.0`.

### Phase 5 — Optional follow-up hardening
These are useful, but not required to begin the release cycle:
- commitlint + husky enforcement
- PR checks workflow
- merge gate workflow
- branch protection rules
